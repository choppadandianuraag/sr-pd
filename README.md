# Smart Rejection: An AI-Powered Intelligent Resume Screening and Constructive Feedback Generation System

---

## 1. Abstract

**Smart Rejection** is an end-to-end AI-powered resume screening system that automates the entire applicant evaluation lifecycle — from resume ingestion and parsing, through multi-dimensional scoring and ranking, to the generation of personalized, constructive feedback emails for borderline applicants. The system is architected as a three-stage workflow pipeline orchestrated by a unified FastAPI webhook server, backed by a PostgreSQL database (Supabase) with pgvector extensions, and integrated with external automation platforms (Make.com) for real-time event-driven processing.

Unlike conventional Applicant Tracking Systems (ATS) that produce binary accept/reject outcomes, Smart Rejection introduces a **three-zone classification model** — *Selected*, *Feedback*, and *Rejected* — where borderline candidates in the *Feedback* zone receive AI-generated, personalized emails identifying specific skills gaps relative to the job description and suggesting actionable improvement paths. The feedback emails are generated **directly by a Large Language Model (LLM)** — specifically Qwen2.5-7B-Instruct hosted via HuggingFace Inference API — by providing the candidate's resume text and job description as context in the prompt, rather than using a Retrieval-Augmented Generation (RAG) pipeline.

---

## 2. Introduction and Motivation

The modern recruitment process is overwhelmed by the volume of applications received per open position. Manual resume screening is time-consuming, inconsistent, and fails to provide candidates with actionable feedback. Existing ATS solutions focus exclusively on filtering, treating applicants as pass/fail data points without the human courtesy of explaining *why* they fell short and *how* they can improve.

Smart Rejection addresses this gap with three key innovations:

1. **Section-Aware Resume Understanding**: Rather than treating a resume as a flat text blob, the system segments resumes into meaningful sections (skills, experience, education, projects, etc.) using regex-based heuristics, enabling more targeted matching.

2. **Hybrid Scoring with Explainability**: The system combines BERT-based semantic embedding similarity (40% weight) with a structured LLM-based ATS score (60% weight) that independently evaluates skills (60%), experience (25%), and education (15%) alignment, producing a transparent, multi-dimensional score.

3. **Constructive Feedback Generation**: Borderline candidates receive personalized rejection emails generated by an LLM that performs a step-by-step analysis — first identifying what skills the candidate *has*, then comparing against the job requirements to find what's *missing*, and finally composing a professional email with specific improvement suggestions.

## 3. System Architecture

The system follows a modular three-workflow architecture, each independently deployable but unified through a master FastAPI webhook server.

### 3.1 High-Level Pipeline

```
Resume (PDF/DOCX/Image)
        │
        ▼
┌─────────────────────────────────┐
│   WORKFLOW 1: PREPROCESSING     │
│  Extract → Segment → Embed     │
│         → Store                 │
└──────────────┬──────────────────┘
               │
               ▼
┌─────────────────────────────────┐
│   WORKFLOW 2: SCORING           │
│  Cosine Similarity (40%)        │
│  + ATS Score (60%)              │
│    ├ Skills (60%)               │
│    ├ Experience (25%)           │
│    └ Education (15%)            │
│  → Zone Classification          │
└──────────────┬──────────────────┘
               │
               ▼
┌─────────────────────────────────┐
│   WORKFLOW 3: FEEDBACK          │
│  LLM-Generated Personalized    │
│  Feedback Emails for           │
│  Borderline Candidates         │
└─────────────────────────────────┘
```

### 3.3 Database Schema

The system utilizes a multi-table relational schema:

- **`applicant_profiles`**: Stores applicant metadata (name, email, phone), raw extracted resume text, segmentation confidence scores, and manual review flags.
- **`applicant_embeddings`**: Stores per-section text (skills, education, work experience, projects, certifications, summary) alongside a single 768-dimensional combined BERT embedding vector.
- **`job_descriptions`**: Stores job metadata, description text, and active status.
- **`job_embeddings`**: Stores embedding vectors for job descriptions using pgvector with IVFFlat indexes.
- **`match_history`**: Audit trail of all scoring operations, recording overall scores, component section scores, weights used, and classification status (selected/feedback/rejected).
- **`feedback_emails`**: Stores generated feedback email content, recipient information, LLM model used, generation status, and delivery tracking.
- **`scoring_config`**: Configurable section-to-section matching weights.

## 4. Methodology

### 4.1 Workflow 1 — Resume Preprocessing Pipeline

#### 4.1.1 Text Extraction

The system supports multi-format resume ingestion through a unified `ResumeProcessor` class with pluggable extractors:

- **PDF files**: Dual-engine extraction using pdfplumber (preferred) with PyPDF2 fallback. If both fail to extract meaningful text (indicating a scanned/image-based PDF), the document is routed to OCR.
- **DOCX files**: Extraction via python-docx with mammoth as fallback for complex formatting.
- **Image files (PNG, JPG)**: Processed through Tesseract OCR, with an optional NuMarkdown-8B-Thinking model for higher-accuracy document understanding (requires GPU).

#### 4.1.2 Section Segmentation

After text extraction, the raw resume text is segmented into structured sections using a **lightweight, deterministic regex-based segmenter** (no LLM dependency). The `ResumeSegmenter` class identifies seven section types:

- Contact Information, Summary/Objective, Work Experience, Education, Skills, Certifications, and Projects

The segmentation algorithm:
1. **Contact extraction**: Scans the first 10–15 lines for email addresses and phone numbers (regex-based), extracting the contact block.
2. **Section header detection**: Each remaining line is evaluated against a library of regex patterns per section type (e.g., `\bwork\s+experience\b`, `\btechnical\s+skills\b`). Formatting heuristics (ALL CAPS, Title Case, colon-ending, markdown bold markers) boost confidence scores.
3. **Confidence scoring**: Each section is assigned a confidence score (0.0–1.0). The overall segmentation quality is assessed; resumes with average confidence below 0.75 or missing critical sections (experience/education) are flagged for manual review.

This design decision — regex over LLM — was made to ensure **determinism, speed, and cost-effectiveness** at scale.

#### 4.1.3 Embedding Generation

A single combined BERT embedding is generated per resume using the `sentence-transformers/all-mpnet-base-v2` model. All section texts (excluding contact information) are concatenated with section-type headers (e.g., `SKILLS:\n<text>`) into a single string, which is then encoded into a **768-dimensional dense vector**. This combined embedding captures the holistic semantic representation of the applicant's profile.

#### 4.1.4 Database Storage

The processed data is stored in a dual-table structure:
- **applicant_profiles**: Stores human-readable data (name, email, raw text, segmentation metadata).
- **applicant_embeddings**: Stores per-section text fields and the combined 768-dim embedding vector.

If an applicant already exists (matched by email), the system performs a soft update — replacing the old embedding while preserving the profile record.

### 4.2 Workflow 2 — Integrated Scoring Pipeline

The scoring pipeline implements a **hybrid scoring model** that fuses semantic similarity with structured requirement matching.

#### 4.2.1 Job Description Processing

When a job description is received:
1. Its text is embedded using the same BERT model (`all-mpnet-base-v2`) for cosine similarity computation.
2. An LLM (Llama 3.3-70B via Groq API) extracts structured requirements in three dimensions:
   - **Skills**: Categorized into *must-have*, *good-to-have*, and *nice-to-have* tiers.
   - **Experience**: Minimum and preferred years, relevant domains.
   - **Education**: Minimum and preferred degree levels, relevant fields.

#### 4.2.2 Scoring Components

Each applicant is scored via two independent channels:

**A. Cosine Similarity Score (40% of final score)**

The cosine similarity between the applicant's combined resume embedding and the job description embedding is computed using sklearn's `cosine_similarity`. This captures broad semantic alignment.

**B. ATS Score (60% of final score)**

The ATS score is computed through LLM-powered structured analysis:

1. **Resume Profile Extraction**: The LLM extracts the applicant's skills list, years of experience, job titles, domains, education level, and field from the raw resume text.

2. **Skills Scoring (60% of ATS)**:
   - For each tier (must-have/good-to-have/nice-to-have), the system computes BERT embedding similarity between each JD requirement and the applicant's skill set.
   - Direct keyword matching provides a bonus (+0.4, capped at 0.85 minimum) for exact skill matches.
   - Tier weights: must-have (50%), good-to-have (30%), nice-to-have (20%).

3. **Experience Scoring (25% of ATS)**:
   - **Years component (40%)**: Score based on whether candidate years meet minimum, preferred, or fall short. Fallback regex extraction from resume text if LLM returns 0.
   - **Domain relevance (60%)**: BERT embedding similarity between candidate's domains/job titles and JD-required domains.

4. **Education Scoring (15% of ATS)**:
   - **Degree level (50%)**: Hierarchical matching (high school → diploma → associate → bachelor → master → PhD) with graduated scoring.
   - **Field relevance (50%)**: BERT embedding similarity between candidate's education field and JD-required fields.

#### 4.2.3 Final Score Computation

```
Final Score = Cosine Similarity × 0.40 + ATS Score × 0.60
```

Where:
```
ATS Score = Skills × 0.60 + Experience × 0.25 + Education × 0.15
```

#### 4.2.4 Zone Classification

After all applicants are scored for a given job, they are classified into three zones using **percentile-based ranking** (not absolute thresholds):

| Zone         | Percentile Range | Action                        |
|-------------|-----------------|-------------------------------|
| **Selected**   | Top 10%         | Advance to interview / next round |
| **Feedback**   | Next 40%        | Receive constructive feedback email |
| **Rejected**   | Bottom 50%      | No further action             |

The `ZoneClassifier` module handles edge cases for small candidate pools (≤5 applicants) and logs the classification into the `match_history` table for audit purposes. Applicants classified as *Feedback* have corresponding records created in the `feedback_emails` table with status `pending`.

### 4.3 Workflow 3 — Feedback Email Generation

#### 4.3.1 Architecture

> **Important Clarification**: The feedback generation system **does not use a RAG (Retrieval-Augmented Generation) pipeline**. The feedback emails are generated **directly by an LLM** — Qwen2.5-7B-Instruct accessed via the HuggingFace Inference API — using a carefully engineered prompt that includes the full resume text and job description as context.

The ChromaDB vector store present in the codebase serves only as a **caching layer** for resume and job description sections. When a feedback email needs to be generated, the system:

1. Fetches the applicant's resume sections from ChromaDB (or lazily syncs from Supabase if not cached).
2. Fetches the job requirements from ChromaDB (or lazily syncs from Supabase).
3. Formats both as structured context strings.
4. Passes them directly to the LLM with the engineered prompt.

This is a **direct LLM generation** approach where the full resume and job description are provided as context in the prompt, not a retrieval-augmented pipeline.

#### 4.3.2 Prompt Engineering

The feedback generation uses a two-part prompt structure:

**System Prompt**: Instructs the LLM to act as a technical recruiter, emphasizing that feedback must be specific to *this particular candidate* and not generic.

**User Prompt (FEEDBACK_EMAIL_TEMPLATE)**: Enforces a structured three-step analysis:

- **Step 1**: Identify 3–5 specific technologies/skills the candidate *has* from their resume.
- **Step 2**: Compare against job requirements and identify what's *missing*.
- **Step 3**: Compose a professional email mentioning the candidate's existing strengths, listing missing skills as improvement areas, and suggesting relevant certifications or courses.

The prompt explicitly instructs the LLM to output *only* the final email (not the analysis steps), and warns against defaulting to generic skill suggestions.

#### 4.3.3 Batch Processing

The `batch_feedback.py` module processes all feedback-zone candidates for a given job:
1. Queries `match_history` for records with `status='feedback'`.
2. De-duplicates by applicant (keeping highest score).
3. Skips applicants who already have generated feedback emails.
4. Generates personalized emails via the LLM for each remaining candidate.
5. Saves generated emails to the `feedback_emails` table in Supabase.

## 5. Master Webhook Server and Automation Integration

The entire system is exposed as a unified REST API via the `master_webhook_server.py` — a FastAPI application that consolidates all three workflow endpoints:

| Endpoint                                 | Method | Description                                    |
|------------------------------------------|--------|------------------------------------------------|
| `/workflow1/process`                     | POST   | Process resume from URL or Base64              |
| `/workflow1/upload`                      | POST   | Process resume from file upload                |
| `/workflow2/score-applicant`             | POST   | Score single applicant against all jobs         |
| `/workflow2/score-job`                   | POST   | Score all applicants for a specific job         |
| `/workflow2/score-all`                   | POST   | Score all applicants for all jobs               |
| `/workflow3/feedback`                    | POST   | Generate feedback for a single candidate       |
| `/workflow3/sync-feedback-emails`        | POST   | Backfill pending feedback email records         |
| `/workflow3/generate-pending-feedbacks`  | POST   | Generate content for all pending feedback emails|

The server integrates with **Make.com** for workflow automation: when a new resume is submitted through a Google Form, Make.com triggers the `/workflow1/process` endpoint, followed by `/workflow2/score-applicant`, and finally `/workflow3/feedback` for borderline candidates — achieving a fully automated, end-to-end pipeline.

## 6. Technical Contributions

1. **Hybrid Scoring Model**: A novel combination of dense semantic embeddings (BERT cosine similarity) with structured LLM-extracted requirement matching across three independent dimensions (skills, experience, education), producing an interpretable and explainable composite score.

2. **Percentile-Based Zone Classification**: Unlike threshold-based systems that produce inconsistent results across varying candidate pools, the percentile-based classification adapts the cutoff boundaries dynamically based on the applicant pool quality.

3. **LLM-Driven Constructive Feedback**: A structured prompt engineering approach that transforms rejection into a constructive experience by generating personalized, skills-gap-aware feedback emails directly through an LLM.

4. **Deterministic Section Segmentation**: A lightweight, LLM-free regex and heuristic-based resume segmenter with confidence scoring and manual review flagging, ensuring consistent and cost-effective preprocessing.

5. **Webhook-Based Real-Time Processing**: Event-driven architecture enabling real-time, automated resume processing through Make.com integration with FastAPI endpoints.

## 9. Conclusion

Smart Rejection demonstrates that AI-powered recruitment systems need not be binary gatekeepers. By combining semantic embedding similarity with structured LLM-based requirement analysis, the system produces transparent, multi-dimensional candidate evaluations. More importantly, by introducing a *feedback zone* and generating personalized improvement suggestions through direct LLM prompting, the system transforms the rejection experience from a dead end into a development opportunity. The three-workflow modular architecture, backed by a robust database schema and webhook-based automation, makes the system production-ready and scalable for real-world deployment.
